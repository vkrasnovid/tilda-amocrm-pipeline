# Stories: Tilda → AmoCRM + Email-цепочка

## Story 1: Webhook-приёмник для Tilda
**Описание:** FastAPI endpoint принимает POST от Tilda при отправке формы (имя, телефон, email). Сохраняет лид в локальную БД.
**Acceptance criteria:**
- [ ] POST /webhook/tilda принимает JSON/form-data от Tilda
- [ ] Валидация: email обязателен, имя обязательно
- [ ] Данные сохраняются в таблицу `leads`
- [ ] Дублирование по email — обновление, не создание нового
- [ ] Ответ 200 + JSON {status: "ok"}
- [ ] Логирование каждого входящего лида
**Агент:** Developer
**Приоритет:** P1
**Оценка:** 2 часа

## Story 2: Создание сделки в AmoCRM
**Описание:** При получении нового лида автоматически создаётся контакт + сделка в AmoCRM через REST API.
**Acceptance criteria:**
- [ ] Создаётся контакт в AmoCRM (имя, телефон, email)
- [ ] Если контакт уже существует (по email) — привязывается к существующему
- [ ] Создаётся сделка в воронке "Воронка" на этапе "Первичный контакт"
- [ ] Сделка привязана к контакту
- [ ] При ошибке API — retry 3 раза с exponential backoff
- [ ] Статус интеграции сохраняется в БД (amocrm_deal_id)
**Агент:** Integrator
**Приоритет:** P1
**Оценка:** 4 часа

## Story 3: Email-цепочка (3 письма)
**Описание:** После создания лида запускается цепочка из 3 писем через SMTP.
**Acceptance criteria:**
- [ ] Письмо 1 "Приветствие" — отправляется сразу после создания лида
- [ ] Письмо 2 "Напоминание" — через 2 дня (48 часов)
- [ ] Письмо 3 "Спецпредложение" — через 5 дней (120 часов)
- [ ] Письма отправляются через Celery задачи с Redis
- [ ] Шаблоны писем в HTML (отдельные файлы)
- [ ] Статус каждого письма в БД (pending/sent/failed)
- [ ] При ошибке SMTP — retry 3 раза
**Агент:** Developer
**Приоритет:** P1
**Оценка:** 4 часа

## Story 4: Остановка цепочки при ответе
**Описание:** Если клиент ответил на любое письмо — цепочка останавливается. Мониторинг через IMAP polling входящей почты.
**Acceptance criteria:**
- [ ] IMAP polling каждые 5 минут проверяет входящие письма
- [ ] Matching по email отправителя с таблицей leads
- [ ] При совпадении — статус цепочки меняется на "stopped"
- [ ] Все pending задачи в Celery отменяются (revoke)
- [ ] Дата остановки записывается в БД
**Агент:** Developer
**Приоритет:** P1
**Оценка:** 3 часа

## Story 5: Telegram-уведомление менеджеру
**Описание:** При остановке цепочки (клиент ответил) менеджеру приходит уведомление в Telegram.
**Acceptance criteria:**
- [ ] Сообщение: "Клиент [Имя] ([email]) ответил на письмо. Цепочка остановлена."
- [ ] Отправка через Telegram Bot API
- [ ] Менеджер ID настраивается через .env
- [ ] При ошибке отправки — логирование, без crash
**Агент:** Integrator
**Приоритет:** P2
**Оценка:** 1 час

## Story 6: Деплой
**Описание:** Docker Compose с FastAPI + Celery + Redis. Health-check, graceful shutdown.
**Acceptance criteria:**
- [ ] docker-compose.yml: app (FastAPI) + worker (Celery) + redis + celery-beat
- [ ] Dockerfile с multi-stage build
- [ ] .env.example со всеми переменными
- [ ] Health-check endpoint GET /health
- [ ] Graceful shutdown для Celery workers
- [ ] Volume для SQLite данных
**Агент:** DevOps
**Приоритет:** P2
**Оценка:** 2 часа

---

**Итого:** 6 stories, ~16 часов разработки
